# 工作流

<important>
禁止使用ssh、scp命令。连接远程服务器、在远程服务器执行命令、上传/下载文件到远程服务器，使用ssh-mcp mcp
</important>

## 任务执行的流程规范

1. 开始任务前，**必须**保证 git 干净，有未提交文件则提交
2. 使用合适的 subagent 执行任务，**必须严格**监督任务完成情况
3. 测试**必须**遵循**测试规范**，测试验证清单通过才算完成任务
4. 更新相关项目文档，包括CLAUDE.md
5. 测试验证完成一个任务/功能后**必须**进行一次 git commit

### 任务启动规范

1. 开发前先确保项目正在运行，否则启动项目

2. 创建回滚条件
**执行初始 git commit**
   - 展示 添加/修改/删除 了哪些文件
   - 使用描述性 commit 消息


### 测试规范

#### 禁止行为

- **禁止**仅凭代码审查就标记功能为 "passed"
- **禁止**只运行单元测试就认为功能完成
- **禁止**使用 curl 测试 API 就宣布功能可用
- **禁止**以任何草率的理由，草率的宣布完成任务

#### 必须行为

完成任务前**必须**彻底测试，保证以下测试清单已完成，才算任务完成

测试验证清单:  
- 代码没有错误
- build 成功
- 简单任务可通过单元测试/代码审查验证
- 对于 web 应用，**必须**通过自动化工具进行端到端浏览器验证，比如 playwright mcp
  1. 使用浏览器模拟用户操作
  2. 验证：
     - 页面正确加载
     - 用户交互产生预期结果
     - 数据正确保存/显示，没有乱码

测试验证通过后，使用 git commit 提交更改: 
- 展示添加/修改/删除了哪些文件
- 使用描述性 commit 消息

### 代码提交规范

**每次代码更改后必须提交**：

```bash
git add -A
git commit -m "feat: 实现用户登录功能

- 添加登录表单组件
- 集成认证API
- 添加登录测试
"
```

为什么：commit 允许回滚错误更改，恢复到工作状态。

## 文件命名规范

## 工具的使用规范

- 使用ssh-mcp上传/下载复杂目录，**必须**用 tar 打包后传输
- 在 WSL 中执行命令的格式: `wsl -d [发行版名称] -- [命令]`
